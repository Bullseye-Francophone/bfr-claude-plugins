# Lint a DCS mission

## Run the linter

POSIX: `"${CLAUDE_PLUGIN_ROOT}/tools/mizlint.sh" all <path> --json`
Windows: `"${CLAUDE_PLUGIN_ROOT}\tools\mizlint.cmd" all <path> --json`

`<path>` = mission project root (contains `src/mission`), a `src/mission` folder,
or a folder of several projects (each is linted separately).
Run a single check with its name instead of `all` (`mizlint list` shows them).
Exit codes: 0 clean, 1 warnings only, 2 errors, 3 execution failure (bad path,
unparseable mission — report the stderr message, do not guess).

## Interpret findings

Findings are JSON records `{check, severity, code, message, file, detail, project}`.
Group by project, then severity (`error` first). For each finding:

1. Explain the consequence in mission-maker terms (what breaks in game or at build).
2. Point at the file and the precise fix.
3. Do NOT apply fixes — mission files are regenerated by the Mission Editor;
   hand-edits can desynchronize `trig`/`trigrules`. Propose, let the user decide.

Background per check, read on demand:
- resources → `${CLAUDE_PLUGIN_ROOT}/knowledge/vanilla/resources.md`
- triggers → `${CLAUDE_PLUGIN_ROOT}/knowledge/vanilla/mission-file.md`
- loading → `${CLAUDE_PLUGIN_ROOT}/knowledge/frameworks/vmct/injection.md`
- flags / names → `${CLAUDE_PLUGIN_ROOT}/knowledge/vanilla/mission-file.md`

## Known traps (read before challenging a finding)

- Key namespace is decided by usage, not by the `DictKey_`/`ResKey_` prefix.
- VEAF numeric keys (102xx–104xx) are NOT stable across missions.
- A mission freshly injected by VMCT but never re-saved in the Mission Editor
  legitimately fails the triggers check — the re-save IS the fix.
- `NAME-GROUP-MISSING` is a warning on purpose: the group may be spawned at runtime.
- Flag linting only covers quoted string flags; numeric flags are not tracked.

## Large output

If findings exceed ~50 entries, summarize counts per code, list all errors
individually, and offer to drill into warnings on request.
